Part 1 · 개념 소개

ABAP Analyzer란 무엇인가

ABAP Analyzer는 SAP 시스템에서 작성된 ABAP 소스 코드를 자동으로 분석하여 품질·성능·보안·표준 준수 문제를 찾아내는 도구다. 사람이 코드를 일일이 읽지 않고도 문제 지점을 기계적으로 짚어 준다.

1.2배경 — ABAP은 왜 분석이 필요한가

  • ABAP(Advanced Business Application Programming)은 SAP ERP(ECC, S/4HANA)의 업무 로직을 개발하는 SAP 고유 언어다.
  • 대기업의 회계·물류·인사·생산 등 핵심 업무 로직 대부분이 ABAP으로 작성되어 있다.
  • 수십 년간 여러 개발자의 손을 거친 방대한 레거시 코드라, 수동 리뷰만으로는 문제를 다 잡을 수 없다.

1.3무엇을 잡아내는가

Performance

SELECT *, 루프 내 DB 조회, SELECT…ENDSELECT, 인덱스 미사용

Quality

미사용 변수, 중복 코드, 과도하게 긴 함수, 오류 처리 누락

Security

SQL Injection, AUTHORITY-CHECK 누락, 위험한 동적 코드 실행

Standard

명명 규칙, 코딩 가이드라인, S/4HANA 비권장 구버전 문법

1.4이미 존재하는 도구 — 그리고 그 한계

도구성격특징
SAP Code Inspector (SCI)SAP 기본규칙 기반 정적 분석의 표준
ABAP Test Cockpit (ATC)SAP 기본SCI 확장, 개발 프로세스 통합

기존 규칙 기반 도구의 한계 규칙에 없는 새 패턴은 못 잡고, 코드의 의도(문맥)를 이해하지 못하며, 사람이 읽을 수 있는 설명·개선안을 자연어로 제시하지 못한다. → 바로 이 지점에서 LLM 기반 접근이 필요하다.
Part 2 · 기획서

LLM 기반 ABAP Analyzer 도구

"ABAP 코드를 넣으면, LLM이 문제를 찾아 자연어로 설명하고 개선 코드(before → after)까지 제시하는 분석 도구." 기존 규칙 기반 도구가 놓치는 문맥 이해와 자연어 설명이 핵심 차별점이다.

2.2목표 & 비목표

Goals

성능·품질·보안 이슈 탐지 + 이유 설명 · 개선 코드 자동 생성 · SCI/ATC 결과 재해석·우선순위화 · 이해하기 쉬운 리포트

Non-Goals (초기)

SAP 실시간 연동 · 코드 자동 커밋/배포 · 문법 100% 완벽 파싱(초기엔 LLM 이해에 의존)

2.4시스템 아키텍처

입력 → 분석 엔진(4단계) → 출력의 3계층 구조.

pipelineflow
┌─ 입력 계층 ─────┐   ┌─ 분석 엔진 ──────────┐   ┌─ 출력 계층 ──────┐
│ ABAP 소스        │→ │ 1. 전처리 / 청킹      │→ │ 이슈 리스트       │
│ 파일·붙여넣기·Git │   │ 2. 규칙 프리체크      │   │ 심각도 / 카테고리 │
│                  │   │ 3. LLM 분석          │   │ 설명 + 개선 코드  │
│                  │   │ 4. 결과 검증          │   │ 리포트 (MD/HTML) │
└──────────────────┘   └──────────────────────┘   └──────────────────┘
  • 1. 청킹 — 긴 프로그램을 논리 단위(FORM/METHOD/FUNCTION)로 분할해 컨텍스트 한도에 맞춤.
  • 2. 규칙 프리체크 — 명백한 안티패턴을 정규식으로 먼저 표시 → LLM 비용 절감 + 정확도 보강.
  • 3. LLM 분석 — 각 청크를 구조화된 JSON으로 분석(부록 A).
  • 4. 결과 검증 — 스키마 검증 + 라인 번호 존재 확인으로 환각 방지.

2.7개발 로드맵

1

PoC

단일 파일 붙여넣기 → LLM 분석 → JSON 리포트. 동작 검증.

2

MVP

청킹 + 규칙 프리체크 + 웹 UI(이슈 목록·개선안). 사용 가능한 도구.

3

확장

다중 파일/프로젝트, 심각도 대시보드, 리포트 export. 팀 활용.

4

통합

Git/CI 연동, SAP 시스템 연동 검토. 파이프라인화.

리스크 & 대응

Appendix A

PoC 프롬프트 실제 설계

ABAP 소스 1개를 입력받아 규칙셋 기준으로 이슈를 찾아 구조화된 JSON으로 출력한다. 프롬프트는 ① System · ② Developer · ③ User 3-블록으로 구성한다.

A.1System 프롬프트 — 역할 · 환각 방지 원칙

system.txtprompt
당신은 20년 경력의 시니어 SAP ABAP 코드 리뷰어입니다.
주어진 ABAP 소스를 정적 분석하여 성능·품질·보안·표준 관점의 문제를 찾습니다.

반드시 지켜야 할 원칙:
1. 실제 코드에 근거해서만 지적한다. 없는 문제를 지어내지 않는다.
2. 각 지적에는 반드시 실제 존재하는 라인 번호를 명시한다.
3. 확신이 없으면 severity를 낮추거나 보고하지 않는다. (거짓 양성 최소화)
4. 개선안(after)은 컴파일 가능한 유효한 ABAP 구문으로 작성한다.
5. 출력은 지정된 JSON 스키마만 사용한다. 그 외 설명 문장을 덧붙이지 않는다.
6. explanation과 suggestion은 한국어로 작성한다.

A.2Developer 프롬프트 — 규칙셋 + 출력 스키마 주입

developer.txtschema
[검사 규칙셋]  PERF-001~006 · QUAL-001~006 · SEC-001~004 · STD-001~003 (부록 B)

[severity 기준]
  high   : 운영 장애·보안 사고 직결 (루프 내 DB조회, SQL Injection)
  medium : 성능 저하·유지보수성 악화
  low    : 스타일·표준 위반

[출력 JSON 스키마] — 이 구조만 사용
{
  "findings": [{
    "id": "규칙ID",  "category": "performance|quality|security|standard",
    "severity": "high|medium|low",  "line": <정수>,
    "title": "<한 줄 요약>",  "explanation": "<왜 문제인지>",
    "suggestion": "<어떻게 고칠지>",  "before": "<문제 코드>",  "after": "<개선 코드>"
  }],
  "summary": { "high": <int>, "medium": <int>, "low": <int> }
}
문제가 없으면 findings=[], summary는 모두 0.

A.5PoC 검증 방법

  • 골든셋 테스트 — 위처럼 "코드 ↔ 정답 JSON" 쌍을 10~20개 준비.
  • 평가 지표 — 탐지율(Recall)·오탐률(False Positive)·라인 번호 정확도.
  • 환각 체크 — 출력 line 값이 입력 코드 라인 범위 내인지 자동 검증.
  • 스키마 검증 — JSON Schema로 필드·타입 강제(파이프라인 자동화).
  • 반복 튜닝 — 오탐 잦은 규칙은 System 원칙에 예외 규정 추가.
